# 古神道v3 P2 収集方式変更提案(方式B: 直接取得+抽出分離)

- 提出: こどたん → くろたん承認待ち(P2設計変更のため)
- 日付: 2026-07-11
- 背景: 承認済みの方式A(Sonnet 5+web検索ツール)が**組織のweb検索利用枠の枯渇**により継続不能

---

## 1. 何が起きたか(事実)

- 2026-07-08にディープリサーチ3本(系統A・検索数百回)+パイロット約60クエリを同日に実行
- 以降、web検索ツールが「利用上限」エラーを返し続け、**2日以上経過した07-11 01:28時点でも回復せず**
- 時間制レート(分/時間)なら回復しているはずであり、**より長い周期の利用枠(または累積上限)に到達**したと判断
- 対策(1本化・60秒間隔・検索回数削減・自動クールダウン)は実装済みだが、枠自体が枯れているため効果なし
- **確定資産は保全済み**: 14県10社+正直な0件(劣化0件は全て隔離済み・汚染なし)

## 2. 方式Aの構造的問題

方式Aは「web検索で探す」ため、1県あたり4〜5回の検索を消費する。47県×10系列=470クエリ+再検証で**2,000回超の検索**が必要になり、今回と同じ枯渇が本走行で再発する構造リスクがある。

## 3. 提案: 方式B「直接取得+抽出分離」

```
Stage 1(検索ゼロ・ローカル実行):
  都道府県神社庁の神社検索DB・神社公式サイトをrequestsで直接取得
  → 生HTMLを保存(Tier1ドメイン優先・文字コード復号込み)
  ※ 秋田県神社庁の「浅間神社」検索URL等、パイロットの調査ノートから
    エントリポイントの実在は確認済み

Stage 2(Sonnet 5・web検索ツール不使用):
  保存済みHTMLを入力として祭神・所在地・逐語引用を構造化抽出
  → クォータ非依存・トークン費のみ

Stage 3(既存のまま):
  Tier分類・祭神検証・重複排除・QA・出典タグ(quote+retrieved_at)
```

### 方式比較

| | 方式A(現行・web検索) | 方式B(提案・直接取得) |
|---|---|---|
| web検索クォータ | 1県4〜5回消費・**枯渇済み** | **ゼロ** |
| コスト | 検索課金+トークン | トークンのみ(HTML抽出は安価) |
| 出典の質 | 検索結果経由(モデル任せ) | **Tier1ソースを最初から狙い撃ち** |
| 生HTML逐語検品 | 別途取得が必要 | **Stage1で取得済み=検品と一体** |
| 事前準備 | 不要 | **47都道府県神社庁のエントリポイント表**(1回だけの作業・こどたんが作成) |
| 弱点 | 枯渇・偽ゼロ | 神社庁DBに未掲載の神社を拾えない(→公式サイト直取得で補完) |

### 副次メリット

- 系統Aの教訓(Shift_JIS幻覚→生HTML逐語照合)と完全整合——**検品用の生HTMLが収集の副産物として自動的に残る**
- 「クォータ逼迫時の偽ゼロ」問題が構造的に消える(検索しないため)

## 4. 承認いただきたい点

1. 方式Bへの切替(パイロット=浅間系の残り33県+0社5県の再検証を方式Bで実施)
2. Stage 1の対象: 47都道府県神社庁DB+パイロット既収集の神社公式URL(Tier1/2準拠・ソース制限は指令書のまま不変)
3. 判定観点・提出物(サンプル20件+収量・棄却率)は方式Aの承認内容を継承

## 5. 既存資産の扱い

方式Aで確定した14県10社は出典検証済みのためそのまま採用。方式Bの結果とマージする。

---

## 6. くろたん承認+条件(2026-07-11・確定仕様)

**方式B承認。** 以下の3条件+新ルールを実装に組み込む:

1. **クローリングマナー必須化**: robots.txt尊重 / アクセス間隔3秒以上 / User-Agent明示(連絡先付き) / 同一ドメインへの連続アクセス制限。「相手は信仰を支える組織のサイト——直接取りに行くからこそ礼儀は検索経由より重い」
2. **47エントリポイント表は実行前にくろたんレビュー**(全走行の入口の質検査)
3. **0件再検証パス+既存資産突合の維持**: 確定14県10社は重複排除して統合。偽ゼロ疑い5県(秋田/北海道/新潟/山梨/兵庫)は方式Bで最優先で洗い直す

**新ルール(指令書運用への追加)**: 大量のweb検索・API消費を伴う走行は、**事前見積もり(概算クエリ数・消費リソース)を指令書に明記し、組織クォータへの影響を申告**する。合算管理の欠落が今回の枯渇の根因。
