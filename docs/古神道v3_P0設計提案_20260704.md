## 3. 提案B: 判定ロジックの実装方針

```
新テーブル: JIKKAN_GUARDIAN_TABLE(core/kojindo.py に追加)
  "甲" → {god_id, god, reading, type, shrine, story_type, headline, template}
  ...10エントリ(対応表は指令書0章のとおり・変更なし)

新関数: resolve_guardian_god_v3(bundle) -> dict
  1. day_master = bundle.shichusuimei.nichikan(指令書指定の四柱推命・日主)
  2. shichusuimei が None(計算失敗時)の場合のみ bundle.sanmei.nichikan にフォールバック
     ※両者は定義上同値(同じ日柱天干)。不一致を検知したらログ出力(異常検知)
  3. JIKKAN_GUARDIAN_TABLE[day_master] を返す。1人1柱・決定的
```

- `calculate_kojindo()` / `calculate_kojindo_v2()` の守護神参照をこの関数に差し替え
- **中央星は削除しない**: `extract_traits()`・体感変換・鑑定プロンプトへの供給は現状維持(指令書P0の「物語素材として残す」に準拠)
- 6龍レイヤー(天中殺)は無変更。十二支レイヤーはP3スコープのため触らない
- v1の `GUARDIAN_GOD_TABLE` は退役後も当面ファイル内に残置(参照ゼロ化のみ)→ 削除はくろたん判断
