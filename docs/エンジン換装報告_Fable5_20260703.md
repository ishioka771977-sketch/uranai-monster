## 2. ログ出力仕様(3-2)

- 全Claude呼び出し(`_call_api` / `_call_api_text`)成功時に、APIレスポンスの `model` フィールドと `stop_reason` をコンソールへ出力(既存の `print("[interpreter] ...")` ログ形式に準拠)。
- リクエストしたモデルとレスポンスのモデルが不一致の場合は `FALLBACK:` マーカー付きで出力し、Streamlit Cloud のログ画面で `FALLBACK` を検索すれば振り替え発生を検知できる。
- `stop_reason` も併記した理由: `max_tokens`(出力切れ)や `refusal`(分類器による拒否)の検知が3-3・下記4章の監視に直結するため。

## 3. max_tokens 報告(3-3)

### 現行設定値

- 呼び出し共通処理: `effective_max = min(max_tokens + 4096, 16000)`(+4096はthinking用ヘッドルーム)
- 各コースの指定値(`_call_api` / `_call_api_text` 呼び出し時):

| 用途 | max_tokens指定 | 実効値(API送信値) |
|---|---|---|
| 深掘り質問・チャート系 | 500〜800 | 4,596〜4,896 |
| 開運暦(日/月/年/大運) | 800〜2,000 | 4,896〜6,096 |
| テーマ別・堕ちる系 | 2,800〜3,200 | 6,896〜7,296 |
| 主要8コース鑑定 | 4,000〜4,500 | 8,096〜8,596 |
| 相性鑑定 | 5,000 | 9,096 |

### Fable 5での注意点(実測前の静的評価)

- Fable 5 は **thinking(思考)が常時オン**で、思考トークンも `max_tokens` の枠内で消費される。現行の +4096 ヘッドルームは Opus 4.7 前提の設計であり、Fable 5 では複雑な鑑定で思考が4096を超え、**本文が途中で切れる(`stop_reason=max_tokens`)リスク**がある。
### 実測結果(2026-07-03 実施・全10呼び出し)

| 項目 | 実測値 |
|---|---|
| stop_reason | **全10回 `end_turn`(切り捨てゼロ)** |
| 入力トークン | 4,441〜11,331(8コース平均 約10,900) |
| 出力トークン(思考込み) | 1,421〜4,990 |
| 最大出力/実効max_tokens比 | 4,990 / 8,596 = **58%**(古神道・最長コース) |

**結論: 現行の max_tokens 設定(+4096ヘッドルーム・上限16,000)は Fable 5 でも不足なし。変更不要を提案する。** 最も長い古神道コースでも42%の余裕があった。今後は追加したログの `stop_reason` を監視し、`max_tokens` が出現したら再検討すればよい。

参考: Fable 5はOpus 4.7より**速く・簡潔**だった(T3同一入力: Opus 82.5s/出力2,996tok vs Fable 47.5s/出力2,155tok)。テスト一式の実費は概算$2程度。

## 4. ⚠️ 重要: フォールバック仕様の訂正(指令書2章の前提について)

指令書には「特定分類器に該当したリクエストは Opus 4.8 へ**自動**フォールバックされる」とあるが、最新のAnthropic API仕様を確認したところ:

- **APIでのフォールバックはオプトイン(明示的な設定が必要)**。自動フォールバックが組み込まれているのはClaudeのコンシューマ向けサービスであり、API直叩きでは `fallbacks` パラメータ + ベータヘッダ `server-side-fallback-2026-06-01` を指定した場合のみOpus 4.8への振り替えが行われる。
- 現行コード(未設定)の場合、分類器に該当すると **HTTP 200 + `stop_reason: "refusal"` + 本文空** が返る。このとき既存コードでは空テキストのJSONパース失敗→例外→**Gemini 2.5 Proへの既存フォールバック**が発動する(Opus 4.8ではなくGeminiに落ちる)。
- 占いコンテンツで分類器(サイバー/バイオ系)に該当する可能性は極めて低いが、指令書の前提と実挙動が異なるため報告する。

**提案(くろたん判断待ち・今回は未実装)**: 指令書の想定どおり「Opus 4.8への自動振り替え」を実現する場合、以下の3行変更で対応可能(2箇所):

```python
response = c.beta.messages.create(          # ← beta に変更
    model=_claude_model_name(),
    max_tokens=effective_max,
    betas=["server-side-fallback-2026-06-01"],   # ← 追加
    fallbacks=[{"model": "claude-opus-4-8"}],    # ← 追加
    system=system_with_time,
    messages=[{"role": "user", "content": prompt}],
)
```

指令書の禁止事項「指示外の変更禁止」に従い、今回のdiffには含めていない。追加したモデル名ログはこの構成でも振り替え検知に機能する(レスポンスの `model` が実際に応答したモデルになる)。

### 価格の確認

指令書記載どおり: Fable 5 = 入力 $10 / 出力 $50 per 100万トークン(Opus 4.8は $5 / $25)。モデルID `claude-fable-5` も確認済み(日付サフィックス不要)。

## 5. プロンプトキャッシュ調査報告(3-4・実装なし)

### 適用可否の結論: **現状の構造のままでは効果が出ない。次フェーズで構造変更が必要**

調査結果(実測ベースの静的分析):

1. **systemプロンプト(基本ペルソナ)は小さすぎる**
   `SYSTEM_PROMPT_BASE` は1,096文字 + 日時コンテキスト約260文字 ≈ 推定1,000〜1,500トークン。Fable 5 の最小キャッシュ単位は**2,048トークン**で、これを下回るプレフィックスは `cache_control` を付けてもサイレントに非キャッシュとなる。→ systemのみへの適用は無効。

2. **理論カード(コース別テンプレート)はuserメッセージ側にあり、人物データが混在**
   算命学5,817字・九星6,077字・数秘5,973字等の大型テンプレートは、冒頭67〜154文字目から人物データ(命式等)が差し込まれる構造。プレフィックス一致方式のキャッシュは最初の差し込み位置で切れるため、そのままではほぼ効かない。効かせるには「固定部(理論カード)を前・可変部(人物データ)を後ろ」に並べ替えるテンプレート再構成が必要 = **鑑定プロンプト本文の改変となり、今回の禁止事項に抵触。次フェーズ案件**。
   例外: 万象学の `BANSHO_SYSTEM_PROMPT`(8,223字・差し込みなし)は system 側にあり単独でキャッシュ可能サイズ。

3. **日時コンテキストの位置は問題なし**
   system先頭の日時情報は「日付」のみで分単位の値を含まないため、日内では安定(キャッシュ破壊要因にならない)。

### 削減見込み

- キャッシュ読み取りは通常入力の約1/10価格($10/M → $1/M)。書き込みは1.25倍。
- 効果が最大化するのは**フルコース/コンプリート鑑定**(1回の操作で8回前後のAPI呼び出しが5分TTL内に連続発生)で、構造変更後なら共有プレフィックス分の入力コストを理論値80〜90%削減できる。
- ただし現行利用規模(月数百円)では削減絶対額は**月数十円〜百円程度**。経営判断上は誤差であり、利用規模拡大時に着手するのが妥当。
